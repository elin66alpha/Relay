# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

Relay is a Flutter client (Android / iOS / Web / desktop) plus a Node backend in `server/` that fronts local CLI agents (Claude Code, Codex, Antigravity `agy`, and experimental OpenCode / Hermes). The client connects only after importing an encrypted credential QR; there is no hard-coded backend URL.

## Commands

The local dev loop is one script — use it after most changes:

```bash
./scripts/build_flow.sh   # flutter analyze+test, node --check all JS files, server tests, build web, pm2 restart, build+install debug APK
```

Run pieces individually when iterating:

```bash
flutter analyze --no-pub
flutter test --no-pub
flutter test test/credential_file_codec_test.dart          # single test file
flutter run                                                 # client against an imported credential
node --check server/server.js                               # server is plain Node, no build; syntax-check only

# Backend (managed by PM2; app names: relay-server, relay-tunnel)
cd server && npm start                                       # or: pm2 restart relay-server --update-env
cd server && npm run credential                              # (re)generate the encrypted credential QR + tokens.json entry

# Web build the backend serves itself (flags matter — see below)
flutter build web --no-pub --pwa-strategy=none --no-web-resources-cdn
```

Two build flags are load-bearing: `--no-web-resources-cdn` bundles CanvasKit locally (the gstatic CDN is unreachable on some networks and leaves the app loaded-but-blank), and `--pwa-strategy=none` disables the service worker so browsers don't serve stale frontend after a backend restart.

## Architecture

**Sessions are keyed by `workdir + agent`, not by device.** This is the central design fact and touches the whole stack:

- Each client stores its own current work directory locally and sends it on every request via the `X-Workdir` header. The server resolves it per-request (`server/lib/workdir.js` `resolveRequestWorkdir`); there is no global "current workdir" state. `RELAY_DEFAULT_DIR` is only the path a brand-new device starts from. Workdirs must be absolute.
- The session scope key is `` `${workdir}\0${agentKey}` `` (SCOPE_SEPARATOR). Two devices in the same path share one conversation, one resumable CLI session, and one history, and mirror each other live. Chat history lives on the backend (`server/lib/history.js`), never on the client — it is reloaded on open.
- Concurrent turns on the same scope are **serialized**, not rejected: `server.js` chains them through per-scope tail Promises (`scopeChains`) because the underlying `claude --resume` CLI session is single-threaded. SSE events carry a `scopeWorkdir` and only reach clients currently subscribed to that workdir.

**Agents** (`server/lib/agents.js`) spawn the real CLI tools. `runClaude` / `runCodex` / `runAgy` / `runOpencode` / `runHermes` each take a per-request `workdir` (falling back to `getDefaultWorkdir()`) — when threading new state through agent execution, pass it via the `runAgent(..., { workdir, settings })` options, not a global. Output streams to the client over SSE (`/api/chat` with `Accept: text/event-stream`). Prompts ride to the CLI as a single argv token, so they are size-capped (`PROMPT_MAX_BYTES`, default 100 KB; Linux caps one argv at ~128 KB).

The agent runners now emit `segment` events for multi-message turns (Claude's mid-task follow-ups, Codex's multiple `agent_message` items). Each segment carries its own timestamp and is rendered as a collapsible block in the frontend. `runOpencode` and `runHermes` use `locateBin()` to find their CLI binary (PATH first, then `~/.opencode/bin/`, `~/.local/bin/`); experimental agents are listed only when detected. The BTW (by the way) sidekick supports Claude, Codex, and Antigravity: Claude forks its native CLI session, while Codex/Agy run a read-only side session seeded with the main Relay transcript. Supported agents use a compact `BTW` text affordance in the Flutter UI.

**Backend layout**: `server.js` (~900 lines) holds config, middleware, shared state, schedulers, and static serving; the API routes live in `server/routes/{meta,push,fs,chat,btw,group,sessions,quota}.js`, each exporting a `createXxxRouter(ctx)` factory that receives shared state/helpers through `routeContext`. A route can only use a helper that is actually present on `routeContext` — when adding endpoints that destructure a new helper from `ctx`, add it to the `routeContext` object too (a missing key is an undefined-is-not-a-function crash at request time, not at startup). New JSON state files should use `server/lib/json-store.js` (in-memory cache + atomic tmp/rename writes, 0o600) instead of hand-rolled load/save; new push notifications go through `server/lib/notify.js`, which fans out to both Web Push and FCM.

**Swarm (multi-agent group chat)** is an orchestration layer over the single-agent pipeline — surfaced to users as "Swarm" (蜂群), but `group` everywhere in code/routes/storage. A group (`server/lib/groups.js`, config in gitignored `groups.json`) is an ordered set of agent members with one canonical transcript under scope key `` `${workdir}\0group:${groupId}` ``. Each swarm **pins its own work tree** (`workdir`, chosen at creation, validated with `validateWorkdir`) and **per-member model/effort/permission** (`memberConfigs`); it is *listed* under the workspace it was created in (ambient `X-Workdir`), so one workspace can hold several swarms on different work trees. `server/routes/group.js` records the human message once, snapshots the transcript, parses `@mentions` (`server/lib/group-turn.js`: `parseMentions`/`deltaSince`/`buildGroupPrompt`), freezes one prompt per summoned member from that snapshot, then runs the member turns with `Promise.all`. The group scope is marked busy for the round, but each member turn passes `concurrencyKey` as its private group session key (`` `${workdir}\0${agentKey}\0${groupId}` ``), so members mentioned in the same human message run in parallel while each member still serializes against its own CLI memory. Three dependency overrides do the group-specific work: `runAgent` resumes the member's own private group session so each keeps separate memory while history lands on the group transcript, `getSettings` returns the swarm's `memberConfigs[agent]` (normalized via `agent-options.js`) instead of the solo-chat `agent-settings.json` store, and `broadcastScope` tags every event with `groupId`. Each member is fed only the delta since it last spoke ("plan B"). The client posts as plain JSON so per-agent deltas mirror to all devices on `/api/events`; see `docs/handbook.md`. The Flutter side is `lib/features/chat/group_chat_{controller,screen}.dart` + `lib/core/models/group.dart`; the left-drawer `SwarmDrawerSection` lists the workspace's swarms as always-visible sub-entries (tap opens directly into one; reloads when `BotChatController.activeWorkdir` changes). The create form picks the work tree (a `/api/workdir/browse` directory picker) and each member's options.

**Client HTTP pipeline**: `lib/core/backend/api_transport.dart` (stores + headers + error mapping) is shared by every API caller — reuse it for new API callers instead of copying connection logic. `MachineCredentialsStore` / `DeviceIdStore` / `WorkdirStore` keep **static** in-memory caches because multiple store instances exist across the app; any new write path must invalidate the cache. Heavy work stays off the UI isolate: credential decryption (PBKDF2 600k iterations) runs in `compute()` on native, and history responses over 256 KB are decoded in an isolate. The shared `/api/events` SSE stream has a 90s idle timeout (3× the server's 30s heartbeat) so silently dead connections hit the reconnect path. Shared UI utils: `lib/core/util/error_text.dart` (`friendlyErrorText`), `format_bytes.dart`, `time_format.dart`.

**Model / Effort / Permission per scope**: `server/lib/agent-options.js` is the single source of truth mapping each agent's selectable model/effort/permission options to exact CLI argv (capability-aware — Agy supports `--model` and permissions, but no effort flag). `buildArgs(agentKey, settings)` returns the argv tokens spliced into each spawn. Selections persist per `workdir + agent` scope in `agent-settings.json` (`server/lib/agent-settings.js`, gitignored) — shared by every device in the scope, like sessions. Endpoints: `GET /api/agent-options`, `GET`+`POST /api/agent-settings`, `GET /api/agent-version`, `POST /api/agent-update` (runs `<cli> update` so newly shipped models become selectable). The client surfaces these in the chat composer's "+" drawer (`lib/features/chat/agent_controls.dart`); the model picker has an "Update CLI" button. Gotchas: Codex permission tiers use `-c sandbox_mode=` (not `-s`, which `codex exec resume` rejects) and must pin `approval_policy=never` (exec is non-interactive, so any approval prompt hangs); Claude has a native `--effort`; brand-new pinned models can be added without code via `server/models-extra.json`.

Experimental agents (opencode, hermes) also appear in `agent-options.js` with their own model/effort/permission trees. opencode models are `provider/model` strings (free models listed statically; paid ones via `models-extra.json`); effort maps to `--variant {minimal,medium,high,max}`; the default permission is `bypass` (non-interactive run would hang on a prompt). Hermes' model comes from its own config.yaml; effort is unset; default permission is `yolo`. Both use `-m <id>` for model selection. The `locateBin()` function in `agents.js` detects the CLI at startup so the app only shows these agents when actually installed.

**File system** is one unified screen (`lib/features/filesystem/file_system_screen.dart`) that browses by **absolute path up to filesystem root**, sets the work path, and uploads/downloads. It uses `/api/workdir/browse` (`listAbsoluteDirectory`); download/upload accept absolute paths (`prepareDownloadAbsolute` / `resolveAbsoluteUploadTarget` in `server/lib/filesystem.js`) and enforce size caps `MAX_DOWNLOAD_BYTES` (300 MB; a folder by its uncompressed total) and `MAX_UPLOAD_BYTES` (100 MB). Folder download streams a `zip` subprocess. Downloads run through an app-level singleton `DownloadManager` (`lib/core/download/`) so progress and the completion notification survive leaving the screen; files save to the system Downloads folder with no picker — Android via a MediaStore `MethodChannel('dev.relay.app/downloads')` in `MainActivity.kt`, desktop via `getDownloadsDirectory`, Web via an anchor click (platform split in `lib/core/platform/file_saver*.dart`).

**Auth & credentials**: protected `/api/*` routes need `Authorization: Bearer <token>` (tokens in `server/tokens.json`); before any token exists they return `TOKEN_NOT_CONFIGURED`. Clients also send `X-Device-Id` and `X-Device-Name`, which the backend records as token last-use metadata. Revoked tokens can be deleted through `POST /api/tokens/:id/delete`. The credential QR is an encrypted envelope (PBKDF2-SHA256 + AES-256-GCM) generated by `server/scripts/create-credential.js`; the plaintext password is never written to disk. Clients import by camera (mobile) or by pasting the payload / uploading the PNG (Web/desktop).

**Backend serves the Web build**: when `build/web/index.html` exists, `server.js` serves it on the API host/port with brotli/gzip compression (explicitly including `application/wasm`) and smart caching (`canvaskit/*` immutable, everything else `no-cache` + ETag).

**Chat sessions, quota watch & diagnostics**: under each `workdir + agent` context there can be up to 8 named chat sessions (`server/lib/chat-sessions.js`); the scope key becomes `` `${workdir}\0${agent}\0${sessionId}` `` and the default `Main` session reuses the legacy context key (no history migration) and can't be deleted. A background quota watcher (`server/lib/quota-watch.js`) polls Claude/Codex 5-hour usage; on a detected reset it broadcasts `quota_reset` over SSE and fires any due **scheduled messages** (`server/lib/quota-schedules.js`) — drafts that auto-send to a session after the reset. These are authored on a dedicated **Scheduled messages** left-drawer screen (`lib/features/quota/quota_scheduler_screen.dart`: one row per claude/codex showing the agent, its next reset time, a message box, Send, and a Clear button when one is queued); the usage dialog (`bot_chat_screen.dart`) is read-only. One schedule may be pending **per source per workdir** (`409 SCHEDULE_EXISTS`, or pass `replaceExisting: true` to overwrite the existing one in place — what the screen's Send does); because a reset is one host-wide event, every workdir's schedule for that source fires on the same reset. The runner serializes through the same `scopeChains` as live turns, marks itself `running` before any `await` (so an overlapping watcher tick can't double-send), reconciles stuck `running` rows to `failed` on startup, and prunes finished rows. Schedule create/cancel/sent/failed broadcast `quota_schedule_*` SSE events (workdir-scoped via `sendEvent`) so other devices in the same workspace re-sync. `GET /api/diagnostics` (`server/lib/diagnostics.js`) returns a host/runtime snapshot for the machine status dialog. `quota-schedules.json` is secret state — do not commit.

**Networking modes** (chosen at setup in `backends/{linux,macos,windows}/`): direct public address, named Cloudflare Tunnel, or Cloudflare Quick Tunnel. `server/ecosystem.config.js` reads `RELAY_TUNNEL_MODE` / `CLOUDFLARED_BIN` / `CLOUDFLARED_ARGS` to decide whether/how PM2 runs the tunnel. Quick Tunnel URLs rotate on restart, so the credential must be regenerated after a URL change.

## Gotchas

- Every APK is debug-signed (no release keystore). An APK built on one machine cannot update an install signed by another; uninstall first with `adb uninstall dev.relay.app` (this clears local data).
- `server/.env`, `server/tokens.json`, `server/chat-history*.json`, `server/credentials/`, `server/quota-schedules.json`, `server/fcm-tokens.json`, and `server/fcm-service-account.json` are environment/secret state — do not commit them.
- The file API (browse/upload/download) refuses sensitive paths (`server/tokens.json`, `.env`, `credentials/`, `~/.ssh`, CLI auth files) and honors the optional `RELAY_FS_ROOTS` allowlist — keep both behaviors intact when touching `server/lib/filesystem.js`.
- Antigravity quota comes from the local `agy` language-server RPC (`RetrieveUserQuotaSummary`). If `agy` is not running and no cached value exists, `/api/usage` reports a clear error instead of guessing.
- Experimental agents (opencode, hermes) are only listed by `GET /api/agents` when their CLI binary is found on the host. `locateBin()` scans PATH then known fallback dirs (`~/.opencode/bin/`, `~/.local/bin/`) with a 60s cache. BTW sidekicks use `btw:<agent>` scope keys and never write to the main session.
